## 目标

提供 TFS Rollback 能力：把某个变更集的改动反做（或回滚到某变更集，反做其后的全部改动），
产生挂起更改由用户审查后签入，对齐 VS 的 Rollback Entire Changeset / Rollback to a Specific Version。
与「获取特定版本」（只动本地、不产生挂起更改）形成互补。

## 实现范围

- core 新增 `rollback(serverPath, mode, changeset)`：`Workspace.rollback` + `RollbackOptions`
  - `single`：仅反做该 changeset
  - `toVersion`：反做该 changeset 之后的全部改动
- server 新增 `POST /api/files/rollback`（参数 `serverPath`、`mode`、`changeset`）
- 前端 API 层补充 `rollback`
- 历史弹窗记录行新增「回滚此变更集…」「回滚到此变更集…」：
  - 确认弹窗说明：只产生挂起更改、不会直接入库、签入前可撤销
  - 对象未映射时置灰并给出原因
- 回滚产生冲突时进入既有冲突弹窗流程
- 完成后刷新挂起更改与目录列表，通知用户到挂起面板审查并签入

## 不在范围

- 不做自动签入（回滚结果必须人工审查）
- 不做跨对象批量回滚（仅当前历史弹窗目标对象范围内）
- 不做三方合并编辑器

## 验收标准

- 历史记录可执行「回滚此变更集…」，确认后产生对应的反做挂起更改
- 历史记录可执行「回滚到此变更集…」，其后全部改动被反做为挂起更改
- 回滚结果可在挂起面板审查、可 undo、可签入
- 回滚遇到本地修改冲突时进入冲突弹窗
- 服务端构建通过、前端 typecheck 通过

### 实际实现内容

- 历史弹窗每条记录右键提供两种回滚：「回滚此变更集」（仅反做该提交）与「回滚到此变更集」（反做其后全部改动，回到该时刻内容）
- 确认弹窗明确：回滚只产生挂起更改、不直接写服务器、可撤销、需签入；执行后顶部通知展示 operations / conflicts / failures 统计并提示去挂起面板审查
- 回滚冲突自动进入既有冲突弹窗；挂起更改可 undo / checkin（复用既有流程）
